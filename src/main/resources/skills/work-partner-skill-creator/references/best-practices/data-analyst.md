# 数据分析最佳实践参考

> 本文件供 AI 工作伙伴在协助数据分析/数据科学岗位工作时参考，作为专业判断的方法论依据。

## 分析工作流程

### 标准五步分析法
1. **明确问题**：分析目的是什么？决策者需要什么？避免"为分析而分析"
2. **数据收集**：确认数据来源、口径定义、时间范围、数据质量
3. **数据清洗**：处理缺失值、异常值、重复数据，核对口径一致性
4. **分析建模**：选择合适的分析方法，从描述→诊断→预测→规范层层深入
5. **输出结论**：结论先行，图表辅助，给出可落地的行动建议

### 问题拆解方法

**MECE 原则**（相互独立、完全穷尽）
- 拆解指标时确保各维度不重叠、不遗漏
- 示例：用户流失原因 = 产品问题 + 价格问题 + 竞品问题 + 服务问题

**指标树拆解**
```
北极星指标（GMV）
├── 用户规模（DAU）× 人均购买（ARPU）
│   ├── 新用户 × 转化率 × 客单价
│   └── 老用户 × 复购率 × 客单价
└── 各渠道贡献拆解
```

**5Why 根因分析**
- 对数据异常连续追问 5 个"为什么"，避免停留在表面现象

## 核心分析模型

### 漏斗分析
```
曝光 → 点击（CTR）→ 注册/加购 → 下单（转化率）→ 支付 → 复购
```
- **定位问题**：找到流失最严重的环节，集中优化
- **分群对比**：不同渠道/用户群/时间段的漏斗差异往往揭示根本原因
- **时序对比**：同一漏斗在不同时间点的变化，用于评估产品改动效果

### 留存分析
| 留存时间窗 | 衡量意义 |
|------------|----------|
| 次日留存 | 产品基础体验，用户第一印象 |
| 7 日留存 | 核心使用习惯是否养成 |
| 30 日留存 | 长期用户价值，是否找到"北极星" |

- 留存曲线趋于平稳（留存基线）说明找到了产品核心价值
- 分群留存：按渠道/行为/属性分析，找高留存用户特征并复制

### 同期群（Cohort）分析
- 按用户注册/首购时间分组，追踪各组随时间的行为变化
- **用途**：识别不同时期用户质量差异，评估产品改动对不同批次用户的影响
- **解读**：同一行下看留存衰减；同一列下看历史批次对比

### A/B 测试规范
1. **实验设计**：单变量原则，随机分流，提前估算样本量
2. **运行时长**：至少覆盖完整周期（7天），消除星期效应
3. **结果分析**：假设检验（t 检验或卡方检验），置信度 ≥ 95%
4. **常见错误**：过早停止实验、多次查看结果导致 p-hacking
5. **结论归档**：每次实验结论留存，形成可查询的实验资产库

### 相关性与因果分析
- **相关 ≠ 因果**：两个指标同步变化不代表存在因果关系
- 验证因果需要：A/B 实验 > 自然实验 > 回归控制变量 > 相关分析
- 遇到"相关性很强"的发现时，必须提醒决策者谨慎下结论

## SQL 常用模式

### 留存率计算
```sql
-- 7日留存：以用户首次活跃日期为基准
SELECT
  DATE(first_active_date) AS cohort_date,
  COUNT(DISTINCT user_id) AS total_users,
  COUNT(DISTINCT CASE WHEN days_since_first = 7 THEN user_id END) AS retained_d7,
  ROUND(
    COUNT(DISTINCT CASE WHEN days_since_first = 7 THEN user_id END) * 100.0
    / COUNT(DISTINCT user_id), 2
  ) AS retention_rate_d7
FROM user_activity_with_cohort
GROUP BY cohort_date
ORDER BY cohort_date
```

### 漏斗转化统计
```sql
SELECT
  COUNT(DISTINCT CASE WHEN event = 'page_view' THEN user_id END)  AS step1_view,
  COUNT(DISTINCT CASE WHEN event = 'add_cart'  THEN user_id END)  AS step2_cart,
  COUNT(DISTINCT CASE WHEN event = 'checkout'  THEN user_id END)  AS step3_checkout,
  COUNT(DISTINCT CASE WHEN event = 'purchase'  THEN user_id END)  AS step4_purchase
FROM events
WHERE dt BETWEEN '2026-03-01' AND '2026-03-16'
```

### 环比计算
```sql
SELECT
  dt,
  metric_value,
  LAG(metric_value, 7) OVER (ORDER BY dt) AS value_7d_ago,
  ROUND((metric_value - LAG(metric_value, 7) OVER (ORDER BY dt))
        * 100.0 / LAG(metric_value, 7) OVER (ORDER BY dt), 2) AS wow_growth_pct
FROM daily_metrics
```

## 数据可视化原则

| 场景 | 推荐图表 | 避免 |
|------|----------|------|
| 趋势变化 | 折线图 | 饼图 |
| 占比构成 | 条形图/饼图（≤5项） | 多层饼图 |
| 分布情况 | 直方图/箱线图 | 折线图 |
| 两变量关系 | 散点图 | 折线图强行连接 |
| 排名对比 | 水平条形图 | 纵向柱状图（标签难读） |

**通用原则**：
- 坐标轴 Y 轴从 0 开始，避免视觉误导
- 一张图只表达一个核心观点
- 用颜色和标注引导读者视线，突出关键数据点
- 图表标题直接写结论，而不是"XX指标趋势图"

## 分析报告结构

### 标准报告框架（结论先行）
```
1. 核心结论（1-3 条，可直接作为决策依据）
2. 数据概览（关键指标当前状态及趋势）
3. 深度分析（按维度拆解，找原因）
4. 问题与机会（异常点和可优化空间）
5. 行动建议（具体、可落地、带优先级和预期效果）
```

### 结论表达规范
- ✅「本月 DAU 同比增长 23%，主要来自短视频渠道新用户，该渠道 7 日留存高于均值 8pp，建议加大投入」
- ❌「本月 DAU 有所增长，短视频渠道表现较好」

## 常见数据陷阱

| 陷阱 | 说明 | 应对方法 |
|------|------|----------|
| 幸存者偏差 | 只分析"留下来"的用户/数据 | 纳入流失/失败样本一起分析 |
| 相关 ≠ 因果 | 同步变化不代表有因果关系 | 用实验验证，明确声明局限 |
| 均值掩盖分布 | 平均值可能掩盖两极分化 | 同时看中位数、P25/P75 分位 |
| 数据口径不一致 | 不同系统同名指标定义不同 | 分析前明确口径，注明来源 |
| 小样本结论 | 样本量不足，结论不稳定 | 标注置信区间，建议积累更多数据 |
| 混淆变量 | 忽略了影响结果的第三个变量 | 分层分析，控制变量 |
| 时序性混淆 | 不同时间的数据混在一起比较 | 严格对齐时间窗口 |
